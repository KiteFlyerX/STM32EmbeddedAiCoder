## 一、项目概述

**EmbeddedAiCoder** 是一款面向 **STM32 开发**的「编码自动化」**桌面工具**(提供 Fluent Design 图形界面)。

它的核心思想是:把 STM32 板子的 debug 输出(串口 UART / SWD·RTT)自动采集并回喂给 AI,让 AI 据此分析问题(崩溃、外设异常、时序错误)并自动编写 / 修改 C 代码,再自动编译、烧录、复采验证,形成闭环,把嵌入式工程师从「烧录 → 看日志 → 定位 → 改码」的机械循环中解放出来。

### 一句话定义
> 一个带 **Win11 风格可视化界面**、能自己「看串口日志、解码 HardFault、改 STM32 代码、自动烧录验证」的 AI 助手。

## 三、要解决的问题(STM32 开发痛点)

STM32 日常开发中存在一个高耗时、低创造性的反馈循环:

1. 编写 / 修改 C 代码 → 用 ST-Link 等烧录固件
2. 板子运行,通过 **串口(UART)** 或 **RTT/SWO** 输出日志,或直接 **HardFault 死机**
3. 工程师在海量日志里**人工定位异常**,或对着一堆 fault 寄存器值**反推崩溃点**
4. 回到代码里**定位并修改**(寄存器配错、HAL 调用顺序、中断、栈溢出……)
5. 回到第 1 步,循环往复

其中「看日志 / 解码 fault → 定位源码行 → 改码」高度模式化,**正是 AI 最擅长接管的环节**;而一个直观的 GUI 让这个循环「看得见、控得住」。

## 五、核心功能模块

| 模块 | 职责 | STM32 / GUI 关键能力 |
| --- | --- | --- |
| **1. 日志采集** | 从板子获取 debug 输出 | UART(pyserial)、RTT/SWO(经 SWD);断线重连、落盘 |
| **2. 故障解码** | 把崩溃变成可定位线索 | HardFault 解析(CFSR/HFSR/MMFAR/BFAR)、栈回溯、`addr2line`→源码行 |
| **3. 过滤与上下文** | 加工成 AI 可用上下文 | 关键字/正则过滤、错误聚类去噪、关联源码裁剪 |
| **4. AI 交互** | 回喂问题,取回 C 代码方案 | Prompt 构建、多轮对话、大模型 API、结构化解析 |
| **5. 代码回写** | C 代码落地工程 | 按文件/函数/行安全替换、备份、Git 提交 |
| **6. 构建** | 自动编译固件 | arm-none-eabi-gcc + Make/CMake;**编译错误回喂 AI 自愈** |
| **7. 烧录与复位** | 自动烧录 / 复位 | pyOCD(默认)/ STM32_Programmer_CLI / OpenOCD |
| **8. 闭环调度** | 串联模块,控制迭代 | 编排、重试/熔断、停止条件、最大轮数 |
| **9. 桌面 GUI** | 可视化呈现与人工介入 | Fluent Design 界面:实时日志、fault 解码、AI diff 预览、状态控制(详见第十二章) |

## 六、典型工作流程

### 流程 A:修复 HardFault / 崩溃(最高频)
1. GUI 连接板子 → pyOCD 烧录并复位;串口/RTT 开始采集
2. 板子崩溃 → 日志页捕获 fault(寄存器值 + 调用栈),故障页自动解码
3. `addr2line` 把 PC/LR 映射到源码行
4. AI 模块拿到「fault 现场 + 源码」,给出根因与修改 → **AI 编码页展示 diff 预览**
5. 一键应用 → arm-gcc 构建 → pyOCD 烧录 → 复采,验证 fault 是否消失
6. 仍存在则带着上一轮结果继续迭代(带最大轮数上限)

### 流程 B:调试外设异常(串口不通 / I2C 无应答)
采集现象 → AI 结合 HAL 初始化 / .ioc 判断 → 改码 → 构建 → 烧录 → 复采验证

### 流程 C:实现新外设功能
自然语言描述 → AI 结合工程结构定位改动 → 生成 C 代码 → 构建 → 烧录 → 自检 → 迭代

| 层 | 方案 | 理由 / 授权 |

## 九、非功能性需求

- **安全性**:AI 固件默认烧录开发/测试板,生产烧录需显式确认;密钥加密存储;回写前自动备份,支持一键回滚
- **稳定性**:串口/SWD 断线自动重连;单轮 AI 失败可重试;**GUI 不因后台长任务卡死(后台线程/异步)**
- **可控性**:最大轮数、超时、熔断;GUI 可随时暂停/中止/介入
- **可观测性**:每步留痕;GUI 实时呈现日志/fault/AI/diff/构建烧录状态
- **响应性**:实时日志高频刷新不卡顿(节流 + 后台线程);diff 预览流畅
- **可移植性**:采集通道/烧录后端/构建工具链通过接口抽象
- **成本控制**:上下文裁剪与缓存

### 12.4 各页面要点

- **② 日志监控页**:`PlainTextEdit` 终端风格;自动着色(`[E]/fault` 红、`[W]` 黄、`[I]` 默认);顶部 `ComboBox`(通道:UART/RTT)+ 搜索 `LineEdit` + 暂停滚动按钮;异常事件自动汇总到顶部。
- **④ AI 编码页**:左侧「对话/任务历史」`ListWidget`,右侧**代码 diff 预览**(新增绿/删除红),底部 `[应用修改] [回滚] [让 AI 重试]`;AI 思考过程用 `StateToolTip` 或流式文本展示。
- **③ 故障诊断页**:fault 寄存器卡片(`CFSR/HFSR/MMFAR/BFAR` 解码为可读含义)、栈回溯 `TableWidget`(地址→函数→**源码行**,可点击跳转)。
- **⑤ 构建烧录页**:构建输出控制台、烧录进度 `ProgressBar`、`Flasher` 后端 `ComboBox`。
- **⑦ 设置页**:用 `SettingCard` 系列(串口/波特率、调试器/烧录后端、构建命令、AI Key/模型、最大轮数/超时),配置与 YAML/TOML 双向同步。
